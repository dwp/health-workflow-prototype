PIP HAS integration

This prototype helps visualise what the referral details area in HAS will look like throughout the stages of integrating with New PIP Service. The prototype will also aid Service Readiness colleagues with preparing relevant training material ready for New PIP Service.

New PIP Service thin slice is due to go live in September 2024.

Prototype v1 (May 2024):
New PIP Service thin slice for Sep 2024

  • See Jira task 2515 for more details (opens in new tab)
  • See Mural board for more details (opens in new tab)
  • This prototype reflects the referral and claimant data that will be displayed in HAS for New PIP Service thin slice integration.
  • Referral and claimant data will be pulled from PIP into HAS via an API. The majority of data fields will not be editable, as HAS will always retrieve the most up to date data from PIP.
  • However, there will be certain data fields which PIP cannot supply and is crucial to HAS, such as 'SREL', 'Date of death', 'Safety measures', and 'Interpreter' information. So these fields must remain editable for now.
  • New data fields that have been added (in addition to what is present in the live environment currently) include PIP Service referral ID, Date of death, GP details, Other specialist or medical contacts, Full name, and Residential address (replacing current postcode field, for PIP thin slice cases only).
  • The 'Preferred name' field has been removed because PIP cannot supply this data for September 2024
  • The intention is, for Septemebr 2024 we display in HAS the most crucial and highest priority data that PIP can supply to us. But overtime we work with PIP to incrementally display more referral and claimant data in HAS.

Additional features

Interruption page alerting users they are moving referral to final status (June 2024)

  • See Jira task 2553 for more details (opens in new tab)
  • See Mural board for more details (opens in new tab)
  • When moving the referral to a 'Done' status (which is one of the final statuses), user will be presented with an interruption page that enables users to pause and reflect. This ensures that users are certain they are taking the correct action when moving the referral into the final status, and understand that it cannot be reversed.
